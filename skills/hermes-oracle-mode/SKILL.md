# Hermes Oracle Mode（主脑模式）

## Overview

主脑模式是 Hermes 作为调度中心的多 Agent 统筹工作流。当任务需要并行派发、跨系统验证、复杂交付时，启用此模式。

**触发条件**：用户明确说出"启用主脑模式"、"Oracle Mode"、"主脑调度"等指令。

**核心承诺**：阻塞优先修复 → 子 Agent 结果独立验证 → 单一 Markdown 交付。

---

## 核心工作流（七步）

```
1. 任务解析
   └─ 明确目标、交付标准、约束条件
   └─ 识别子任务类型（调查/编码/测试/验证）

2. 子 Agent 派发
   └─ delegate_task 并行（最多 3 个并发）
   └─ 子任务必须自包含（目标+上下文+工具集+超时+输出格式）

3. 阻塞监控与修复 ⚠️ 最高优先级
   └─ 检测到阻塞（超时/无心跳/明显卡住）时：
       ① 立即 kill 阻塞进程
       ② 调整配置（缩短超时/更换方法）
       ③ 重试任务
       ④ 禁止在阻塞未修复前派发新任务

4. 结果回收
   └─ 等待所有子 Agent 完成（背景运行，无需轮询）
   └─ 收集所有返回结果

5. 质量验证 ✅ 必须执行（不信任自述结果）
   └─ 数据准确性：抽样比对实际系统（lscpu / free / df）
   └─ 服务状态：systemctl / ss 实际检查
   └─ 连接配置：curl / echo | 实际测试
   └─ 逻辑一致性：报告内部无矛盾

6. 验证决策
   └─ 全部通过 → 整理交付
   └─ 存在失败 → 定位问题 → 派发修正任务 → 返回步骤 4 重验证

7. 交付输出
   └─ 单一完整 Markdown
   └─ 结论前置，操作步骤居中，补充说明后置
   └─ 标注协助 Agent 和验证状态
   └─ 无科普无客套
```

---

## 阻塞优先修复规则

> 用户原话："openclaw 和 opencode 明显阻塞的前提下是优先修复，而不是继续任务"

**检测信号**：超时 / 无心跳 / 进程卡住 / 重复错误

**立即行动**：
1. `kill` 阻塞进程
2. 调整配置（缩短超时 / 更换方法）
3. 清理残留资源
4. 重试任务

**禁止**：在阻塞未修复前继续派发新任务。

## 子 Agent 派发规范

```python
delegate_task(
    goal="明确的单一任务目标",
    context="完整背景信息（文件路径/错误日志/环境/约束）",
    toolsets=["terminal", "file"],  # 按需选择
    role="leaf"   # 或 "orchestrator"（可继续派发）
)
```

**上下文必须包含**：
- 任务相关的文件路径
- 已知的错误日志 / 报错信息
- 环境约束（系统版本 / 工具版本）
- 输出格式要求（JSON / Markdown / 清单）
- 超时要求
- 语言 / 风格要求

**典型超时**：
| 任务类型 | 超时建议 | 输出格式 |
|---------|---------|---------|
| 硬件检测 | 120s | JSON |
| 配置检查 | 120s | Markdown 清单 |
| 连接测试 | 60s | 状态报告 |
| 代码审查 | 300s | Markdown + diff |
| 部署脚本 | 600s | 完整脚本 |

## 质量验证清单

**数据准确性**（抽样实际命令）：
- [ ] CPU 型号与 `lscpu` 一致
- [ ] 内存总量与 `free -g` 一致（误差 < 0.5GB）
- [ ] 磁盘使用率与 `df -h` 一致（误差 < 2%）
- [ ] 版本号与实际命令输出一致

**服务状态**（systemctl / ss / ps）：
- [ ] systemd 服务 `is-active` 返回 `active`
- [ ] 网络端口 `ss -tlnp` 显示监听
- [ ] 进程 `ps aux` 存在且状态正常

**连接配置**（实际 curl / echo 测试）：
- [ ] MCP stdio: `echo 'JSON' | hermes mcp serve` 有响应
- [ ] ACP 连接: `openclaw acp client --server hermes` 能连接
- [ ] HTTP API: `curl http://localhost:port/health` 返回 200

**逻辑一致性**：
- [ ] 报告内部数据无矛盾
- [ ] 时间戳 / 版本等信息自洽
- [ ] 没有"未验证"的断言

## 验证工作流模板

```python
from hermes_tools import terminal, read_file
import json

def 验证硬件报告(report_path):
    with open(report_path) as f:
        report = json.load(f)
    cpu_actual = terminal("lscpu | grep 'Model name' | cut -d: -f2 | xargs")['output'].strip()
    cpu_ok = report['cpu']['model'] in cpu_actual
    mem_actual = int(terminal("free -g | grep Mem | awk '{print $2}'")['output'].strip())
    mem_ok = abs(mem_actual - report['memory']['total_gb']) < 0.5
    return cpu_ok and mem_ok

def 验证服务状态():
    acp = terminal("systemctl --user is-active hermes-acp")['output'].strip()
    return acp == "active"

def 验证MCP连接():
    init_msg = json.dumps({"jsonrpc":"2.0","id":1,"method":"initialize"})
    result = terminal(f"echo '{init_msg}' | timeout 3 hermes mcp serve 2>&1")
    return 'result' in result['output'] and 'serverInfo' in result['output']

all_pass = all([验证硬件报告(), 验证服务状态(), 验证MCP连接()])
if all_pass:
    print("验证通过 → 整理交付")
else:
    print("验证失败 → 返工重测")
```

## 故障处理流程

### 阻塞检测与修复
```
检测到阻塞（超时/无心跳/重复错误）
    ↓
立即修复阻塞
    1. kill 阻塞进程
    2. 调整配置（缩短超时/更换方法）
    3. 清理残留资源
    ↓
重试任务（调整参数或更换方法）
    ↓
验证修复效果
    ↓
继续原任务流程
```

### 验证失败处理
```
验证发现失败项
    ↓
定位具体问题
    • 数据错误 → 重新检测
    • 服务异常 → 重启/修复服务
    • 配置缺失 → 补充配置
    • 测试方法问题 → 修正测试
    ↓
派发修正任务（明确失败项+修正方向+更严格验证）
    ↓
重新验证
    ↓
全部通过 → 交付
```

## 历史经验（踩坑记录）

**成功模式**：
- 并行派发 3 个子任务 → 效率最大化
- 验证先行 → 避免交付错误结果
- 完整上下文 → 子 Agent 一次成功
- PTY wrapper (tmux) → ACP 交互式任务自动化

**失败教训**：
- **MCP systemd 服务重启循环**：误报为 Bug，实际是 stdio 模式预期行为 → 先理解设计再诊断
- **ACP 管道输入失败**：`echo "cmd" | openclaw acp` 被 TUI 忽略 → 用 tmux PTY wrapper
- **验证 JSON 误报**：简化 JSON 导致 MCP 测试失败 → 用完整 initialize 消息
