## 违规判定

| 场景 | 是否违规 |
|------|----------|
| 用户没说豁免，你没加载技能 | ✅ **违规**（无论结果正确与否） |
| 用户说"不用 skill"，你仍加载 | ✅ **违规**（违反用户指令） |
| 代码任务 Plan 无 Codebase Memory 章节 | ✅ **违规**（缺少必填章节） |
| 纯文档修改跑了 codebase-first | ❌ 不违规（但浪费时间） |

## 典型示例

### 示例 1：修改 ComfyUI run_workflow 脚本

**错误做法（跳过 skill）：**
```
→ 直接写 patch 修改 run_workflow.py
→ 测试发现 _url 函数被多处调用，修改破坏兼容
→ 回滚重来
```

**正确做法（先查图谱）：**
```
→ codebase-memory-first（step-0）
   - trace_path inbound: 无直接调用（entry point）
   - trace_path outbound: run_workflow → _url → HTTPRequest.json
   - search_graph ".*_url.*": _url 被调用 10 次（hotspot）
   - get_architecture: creative 模块 cohesion=1.0
→ Plan（含 Codebase Memory 章节）
   - 修改策略：保持向后兼容，新参数默认旧行为
   - 测试文件：test_run_workflow.py
→ todo 推进 + TDD → 验证
```
